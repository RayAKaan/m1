# v0.6 Definition of Done Checklist

**Project:** M1 (Mn)
**Version Target:** v0.6 "Credible Prototype"
**Last Updated:** 2026-06-13

Use this as the single source of truth when deciding "are we ready to call it v0.6?"

---

## Core Semantic Requirements (Must Have)

- [ ] **Phase Graph is integrated into `m1c.m0`**
  - `was x.Phase` queries call the real Phase Graph
  - At least some `was` results are constant-folded at compile time (0 or 1 literals in generated C)
  - `freeze` and phase recording for live variables go through the Phase Graph

- [ ] **`now` invariants are re-checked after mutation (Q1 fixed)**
  - Assignments to variables with active `now` constraints emit runtime checks
  - The classic bank balance / invariant corruption examples now behave correctly

- [ ] **Self-hosting is stable enough**
  - `m1c` can successfully compile `src/m1/m1c.m0` and produce a working binary
  - The resulting binary can at least parse and run non-trivial programs

- [ ] **Bare expression statements work (Q5)**
  - `say("hello");` is valid as a statement without `set _ : Int = ...`

- [ ] **All 13 existing temporal operator tests still pass**

---

## Completeness of Temporal Operators

- [ ] Basic `will` recording and return guards work reliably
- [ ] At least simple `will` + `now` conflict detection produces a warning (Q3)
- [ ] `when` triggers continue to re-evaluate correctly on mutation

---

## Tooling & Portability

- [ ] The M0 bootstrap (m0c) builds cleanly on Linux (and preferably macOS)
- [ ] A basic `m1 repl` exists (even if minimal)
- [ ] `m1 check`, `m1 run`, and `m1 build` work on at least one non-Windows platform

---

## Evidence & Documentation

- [ ] `paper/evidence/test-results.md` has been updated with current status
- [ ] At least some golden `_test_*.c` outputs have been regenerated with the new integrated compiler
- [ ] `README.md` "Known Limitations" section accurately reflects v0.6 reality and links to this roadmap
- [ ] The 5 gap issues created during the audit are updated (closed, or clearly marked as "addressed in v0.6" or "deferred to M2")

---

## Process & Release

- [ ] `VERSION` file updated to `0.6.0`
- [ ] `CHANGELOG.md` has a proper v0.6 section
- [ ] A git tag `v0.6` is created
- [ ] The CI workflow (`.github/workflows/test.yml`) passes on the main branch

---

## Nice-to-Haves (Strongly Desired but Not Blockers)

- [ ] At least 10–15 non-test `.m1` example programs exist in the repo
- [ ] Basic syntax highlighting is available (VS Code or tree-sitter)
- [ ] Error messages have improved for at least the temporal operators
- [ ] A short "Writing Real Code in M1" guide or expanded language tour exists

---

## How to Use This Checklist

1. Keep this file in the repo root.
2. Before any "v0.6" release candidate, go through every item and mark it honestly.
3. Items that are intentionally deferred should be moved to a "M2 / Future" section.
4. When the majority of "Must Have" items are checked, you are ready to cut a real v0.6.

**Remember:** v0.6 does not need to be perfect. It needs to be believable — the gap between what the paper claims and what the compiler actually does should be dramatically smaller than it was in v0.5.
