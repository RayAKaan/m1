# M1 (Mn) — v0.6 Roadmap: "Credible Prototype"

**Goal:** Make the claims in the paper actually true in the delivered artifact.

**Current Status (v0.5):** Strong idea + good research packaging, but the self-hosted compiler does not implement several core temporal features that the paper presents as working. The Phase Graph is a standalone prototype, self-hosting is broken, and there are known semantic gaps.

**Target State (v0.6):** A self-hosted compiler that correctly implements the five temporal operators as described, with the Phase Graph integrated, `now` working properly, and the toolchain in a state where serious evaluation and further development become realistic.

**Timeline (rough):** 3–6 months of focused work (solo or small team).

## Guiding Principles for v0.6

1. **Claims must match reality** — If the paper says something works (especially Phase Graph + temporal operators), the compiler that ships with v0.6 must actually do it.
2. **Honesty over hype** — Any remaining gaps must be clearly documented and tracked.
3. **Usability over completeness** — Prioritize making the existing model correct and usable rather than adding lots of new features.
4. **Integration over prototypes** — Standalone C modules are acceptable for exploration, but by v0.6 the important logic must live inside the self-hosted compiler (`m1c.m0`).

## Phase 1: Core Semantic Correctness (Highest Priority)

### 1.1 Phase Graph Integration (The #1 Task)

**Why this matters:** This is repeatedly described as the central technical contribution of the project.

**Current State:**
- Phase Graph exists as a high-quality standalone C implementation (`src/m1/phase_graph.c` + `.h`, ~637 lines total).
- The self-hosted compiler already declares the external interface (`phase_graph_init`, `phase_graph_record`, `phase_graph_freeze`, `phase_graph_query`, `phase_graph_will_check`) but these are not wired into the actual type checker or code generator of `m1c.m0`.

**Definition of Done for v0.6:**
- `was x.Phase` queries are folded to compile-time literals (0/1) when statically decidable, using the Phase Graph.
- `freeze`, `will`, and phase recording are properly hooked into the live variable tracking inside the self-hosted compiler.
- The golden test outputs in `paper/evidence/` can be regenerated by the integrated system (not just the standalone module).
- All 13 temporal operator tests still pass, and new tests cover the integrated behavior.

**Suggested Implementation Order:**
1. Study how the standalone `phase_graph.c` is currently called from the golden C output (look at generated `_test_*.c` files from before cleanup).
2. Add proper AST nodes / lowering for temporal constructs inside `m1c.m0` (if not already present).
3. Implement a thin M0 wrapper layer that calls the external phase graph functions at the right moments (declaration, assignment, freeze, will, was).
4. Modify the code generator in `m1c.m0` to emit calls to these wrappers instead of (or in addition to) the current ad-hoc logic.
5. Wire `was` queries to call `phase_graph_query` and fold the result when possible.
6. Add a compile-time path that can prove simple `was` facts without runtime calls (the main value of the Phase Graph).
7. Write integration tests that would have failed before but now pass because of the real Phase Graph.

**Risks:**
- Performance / stack usage in the interpreter (may interact with the self-hosting blocker).
- Keeping the standalone C version and the M0-integrated version in sync during transition.

**Owner:** Primary technical owner of the compiler.

### 1.2 Fix `now` Re-check After Mutation (Q1)

**Current Bug (from `paper/evidence/test-results.md`):** `now` invariants are only checked at declaration. Mutations after the `now` declaration do not re-evaluate the invariant.

**Definition of Done:**
- Every assignment to a live variable that participates in one or more active `now` invariants must emit a runtime check (or a static proof).
- Existing `now` tests must continue to pass.
- New tests must demonstrate re-checking after mutation (e.g. the bank balance example that currently only checks once).

**Implementation Notes:**
- This is mostly in the handling of `set` / assignment for live variables.
- Start simple: always emit a check on assignment for variables that have active `now` constraints.
- Later add more sophisticated static analysis to elide checks when possible.

**Priority:** Very high — directly undermines the "temporal logic" value proposition.

### 1.3 Self-Hosting Stability (Stack Overflow Fix)

**Current Blocker:** `m1c.exe` compiling `src/m1/m1c.m0` (~1,098 lines) fails with stack overflow (`0xC00000FD`).

**Definition of Done (minimum for v0.6):**
- `m1c` can successfully compile its own source (`m1c.m0`) and produce a working binary.
- The resulting binary can at least parse and type-check non-trivial programs.

**Possible Approaches (pick one or combine):**
- Increase interpreter stack.
- Reduce recursion depth in the parser / checker / codegen (rewrite hot paths iteratively where possible).
- Chunked / staged compilation.
- Hybrid bootstrap — fall back to `m0c` for very large files during transition.

**Priority:** High for credibility. A self-hosting language that cannot self-host is hard to take seriously.

---

### 1.4 Bare Expression Statements (Q5)

**Current Pain Point:** Everything must be wrapped in `set _ : Int = expr;`.

**Definition of Done:**
- Top-level expressions in blocks are valid statements (their value is discarded).
- Existing code that uses the `set _` workaround continues to work.
- Parser, checker, and codegen updated.

**Effort:** Medium. Mostly parser and lowering work.

## Phase 2: Completeness of Temporal Operators

### 2.1 `will` + `now` Conflict Detection (Q3)

**Goal:** Statically detect (or at least warn about) contradictory combinations of `will` and `now` on the same variable.

**Definition of Done:**
- At least a warning (preferably an error) for obvious contradictions.
- Initially conservative (only catch simple cases).

**Leverage:** The Phase Graph work from 1.1 should make this much easier.

### 2.2 Improve `will` and `when` Semantics
- Ensure `will` return guards are reliable and not bypassed.
- Make `when` triggers properly re-evaluate on relevant mutations.
- Add better scoping / lifetime rules for triggers and commitments.

## Phase 3: Developer Experience & Portability (Parallelizable)

### 3.1 Cross-Platform Toolchain
- Make the full bootstrap (m0c) + self-hosted (m1c) build and run cleanly on Linux and macOS.
- Reduce or eliminate Windows-specific assumptions.
- Update `tests/build/run_tests.ps1` or create a cross-platform equivalent.
- **Stretch:** Provide pre-built binaries for the three major platforms.

### 3.2 Basic Tooling
- Syntax highlighting (VS Code, Neovim).
- A simple tree-sitter grammar.
- A basic REPL (`m1 repl`).
- `m1 check` should be fast and give good error messages.

### 3.3 Error Messages & Diagnostics
- Finish the M10xx (compile) / M11xx (runtime) error code scheme.
- Improve location reporting and messages for temporal errors.

### 3.4 Ergonomics & Stdlib
- Support for bare expressions (already in Phase 1).
- More natural syntax for common patterns.
- Expand the standard library *after* the core temporal semantics are solid.

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Phase Graph integration harder than expected | Medium | High | Prototype in a branch early. Simplify initial version. |
| Self-hosting fix requires major interpreter changes | Medium | High | Hybrid approach (use m0c for the compiler itself during v0.6). |
| `now` re-check interacts badly with `when` triggers | Low-Medium | Medium | Design live-variable tracking carefully from the start. |
| Loss of momentum after initial burst | High | High | Set very small, shippable milestones. Celebrate each one. |
| Platform work takes longer than expected | Medium | Low | De-prioritize until core semantics are solid. |
